## 一、软件概述

### 1.1 软件简介

Luxsin AI 音频设备智能助手系统是一款面向 Luxsin 系列音频硬件设备的智能交互与管理软件。本软件通过集成大语言模型能力，为用户提供自然语言驱动的音频设备控制、参数化均衡器（PEQ, Parametric Equalizer）智能优化、设备配置管理、对话式使用指导等一体化服务，实现"用自然语言操控专业音频设备"的交互体验。

系统采用前后端分离架构，后端基于 Python FastAPI 异步框架，通过 Server-Sent Events（SSE）协议向前端实时推送 AI 响应；前端以浏览器为载体，直接与局域网内的 Luxsin 硬件设备通信，完成设备参数读取和写入操作。后端仅与 AI 服务、数据库、Redis 交互，不接触设备，既保证了用户设备的网络隔离安全性，又避免了后端服务器对用户家庭网络的直接访问。

### 1.2 软件用途

1. **面向终端消费者**：为普通用户提供对 Luxsin 音频设备的自然语言控制能力，降低专业音频调校门槛；
2. **面向发烧友与音频工程师**：提供 PEQ 智能优化、A/B 曲线对比、历史回滚等高级功能，辅助音频调校决策；
3. **面向设备厂商**：作为 Luxsin 设备的配套软件，提供统一的人机交互入口和使用数据采集分析入口。

### 1.3 软件主要功能

- 基于大语言模型的多轮对话式交互
- 设备参数实时查询与变更（音量、输入源、输出源、显示、语言等 30+ 项）
- 参数化均衡器（PEQ）预设管理：读取、创建、修改、删除
- 均衡器智能优化：由 AI 生成最优 PEQ 滤波器组合
- 均衡器 A/B 曲线对比可视化（基于 Canvas 绘图）
- 优化结果的一键应用与回滚
- 多会话管理：自动命名、历史保存、上下文压缩
- 多语言支持：简体中文、繁体中文、英语
- Token 使用量实时统计与轮询显示
- 工具调用结果的会话级别持久化

## 四、功能详细说明

### 4.1 用户认证与设备识别

系统以用户设备的 **MAC 地址**为身份标识。浏览器端通过访问同局域网内的 Luxsin 设备获取 MAC，并作为后续所有会话/消息/统计查询的筛选条件。同一台设备的所有会话归属于同一用户命名空间。

### 4.2 多轮对话与流式响应

- 对话端点：`POST /sse/question`，请求体包含 `question`（用户输入）、`mac`、`language`、`device`、`chat_id`（可选）；
- 响应为 SSE 协议流，每个事件为 `data: <JSON>\n\n`，JSON 的 `type` 字段取值为：
  - `text`：文本片段，前端实时追加渲染；
  - `tool_use`：AI 要求调用前端工具，前端执行后通过 `/chat/tool_result` 回传；
  - `error`：请求失败；
  - `done`：本轮对话结束；
- 对话编排器内部维护 `while True` 循环，用于支持 AI 连续多次工具调用直至任务完成；
- 当 Claude 返回 `stop_reason=tool_use` 时，自动分发至后端或前端工具执行器；完成后把 `tool_result` 加入消息上下文并进入下一轮 AI 调用；
- 当 `stop_reason=end_turn` 时，触发可选的背景任务：上下文压缩（消息数或 token 超阈）和会话标题生成（首轮对话）。

### 4.3 会话与消息管理

| 接口 | 方法 | 功能 |
|---|---|---|
| `/sse/chats` | GET | 按 MAC 查询未删除的会话列表 |
| `/sse/messages` | GET | 按 chat_id 查询消息历史（默认消息 + 带 PEQ 的优化消息） |
| `/sse/clear` | POST | 软删除会话（`deleted=True`，保留数据） |
| `/sse/recent_consumption` | GET | 统计 MAC 近 4 小时的 token 消耗 |
| `/sse/optimization_records` | GET | 查询会话内的 EQ 优化记录 |
| `/sse/update_message_applied` | POST | 更新优化消息的应用/回滚状态 |
| `/chat/tool_result` | POST | 接收前端工具执行结果（经 Redis 转发给等待协程） |
| `/health` | GET | 健康检查端点 |

### 4.4 AI 工具调用系统

系统把 AI 可调用的工具分为两类，由 `_executor` 字段区分：

**后端工具（`_executor=backend`）**：由服务端直接执行，无需浏览器参与。

- `optimize_eq`：调用 AI 以结构化 JSON Schema 输出优化后的 PEQ 参数（包括 10 条滤波器、前置增益、可删除标志、自动预设标志），结果写入数据库并关联 `before_peq`/`after_peq` 字段。

**前端工具（`_executor=frontend`）**：需浏览器直接访问用户局域网内的 Luxsin 设备执行。

- `get_device_settings`：读取设备全量配置；
- `set_device_settings`：写入设备配置（音量、语言、输入源、输出源、显示模式等）；
- `get_peq_list`：获取设备上已存 PEQ 预设名称列表；
- `get_current_peq`：读取当前激活 PEQ 参数；
- `set_peq`：创建或覆盖 PEQ 预设；
- `delete_peqs`：批量删除 PEQ 预设。

前端工具调用流程采用**后端挂起等待 + Redis Pub/Sub 通知**的机制：

1. 后端在 SSE 流中推送 `type=tool_use` 事件，指示前端执行；
2. 后端协程订阅 Redis 通道 `tool_result:<tool_use_id>` 并异步等待（默认超时 30s）；
3. 浏览器执行工具后通过 `POST /chat/tool_result` 上报；
4. 后端处理端点把结果 `SET` 到 Redis Key 并 `PUBLISH` 通知；
5. 原等待协程收到通知后取出结果，继续对话循环；
6. 超时未响应时自动补偿一个 `is_error=true` 的占位结果，防止对话阻塞。

### 4.5 参数化均衡器（PEQ）智能优化

**优化流程：**

1. 用户在对话中表达需求（如 "我觉得低频不够"），AI 判断需要优化 EQ；
2. AI 调用 `get_current_peq` 前端工具读取设备当前 EQ；
3. AI 调用 `optimize_eq` 后端工具，后端使用专用系统提示词 `AI_EQ_OPTIMIZE_PROMPT` 请求 Claude 以结构化 JSON 输出优化方案；
4. 后端将优化前（`before_peq`）与优化后（`after_peq`）同时持久化为一条 `OPTIMIZING` 类型的消息；
5. 前端在历史消息中检测到 `type=OPTIMIZING` 且 `before_peq` 与 `after_peq` 均不为空的消息，渲染为"优化卡片"，提供两个操作按钮：
   - "View comparison"：显示 A/B 频响曲线对比图（Canvas 绘制）；
   - 依据 `applied` 状态动态显示 "应用" 或 "回滚" 按钮，点击后触发 `set_peq` 设备操作并调用 `/sse/update_message_applied` 同步状态。

**频响曲线算法（简化版）：**

对 20Hz–20kHz 区间内的 220 个对数等分频点，累加每个滤波器在该频点处的高斯近似增益，最终输出点集由前端 Canvas 逐段连线绘制。横轴为对数频率，纵轴为增益（dB），并绘制网格与坐标轴标签。

### 4.6 上下文压缩与摘要

长对话会导致每轮请求 token 数不断膨胀。系统采用**可配置阈值触发的异步摘要**机制：

- 阈值配置：`SUMMARY_MAX_TOKENS`（默认 10000）、`SUMMARY_MAX_MESSAGES`（默认 21）；
- 在一轮对话 `end_turn` 后检测累计 token 或消息数是否超阈；
- 若超阈，以 `BackgroundTasks` 方式异步调用 `compress_context`，使用专用摘要提示词请求 Claude 生成摘要；
- 摘要作为新消息以 `type=SUMMARIZE` 持久化；
- 下次拉取上下文时，`get_content_messages` 仅取"最后一次摘要及其之后"的消息，前置增加合成对话 `[历史对话摘要] ... / 已了解历史上下文，继续为您服务`，从而让模型延续对话而无需回放全部历史。

### 4.7 会话标题自动生成

首轮对话结束后（`chat.title == "New Chat"`），系统异步调用 Claude 根据用户输入内容生成不超过 10 字的精炼标题，并通过 `update_chat_title` 落库。支持按语言（简/繁/英）选择对应提示词。

### 4.8 Token 消耗统计

- 每条 `Message` 记录 `tokens` 字段（输入+输出总和）；
- `/sse/recent_consumption` 接口按 MAC + 最近 N 小时聚合求和；
- 前端以 20 秒周期轮询，并在工具栏实时显示。

### 4.9 消息完整性修复

由于网络中断或前端关闭等原因，可能出现 `assistant` 发起了 `tool_use` 但没有对应 `tool_result`、或 `user` 消息没有 `assistant` 回复的"断裂"情况。`repair_messages` 函数在载入上下文时执行：

- 检测 assistant 消息中的 `tool_use_id` 集合，比对下一条 user 消息的 `tool_result_id` 集合；
- 对于缺失的 `tool_use_id`，补齐一条 `is_error=true` 的占位 `tool_result`，保证后续请求 Claude 时协议合法；
- 对于用户消息后 AI 没有回复的情况，插入"（回复丢失）"占位。

### 4.10 Luxsin 设备通信协议

设备通信采用基于 HTTP 的定制协议：

- GET 接口：`http://<设备IP>/dev/info.cgi?action=<action>`，响应体经自定义 Base64 编码；
- POST 接口：写入类操作（PEQ 保存、删除）对请求体进行自定义 Base64 编码后 `application/x-www-form-urlencoded` 提交；
- 编码算法：标准 Base64 基础上做字符替换，实现与设备固件兼容的轻量级数据封装；
- 超时控制：3 秒；
- 错误处理：统一抛出业务异常并在前端转化为用户可读错误信息。

## 五、安装与部署

### 5.1 容器化部署（推荐）

**前置条件：**
服务器安装 Docker 20.10+、Docker Compose V2；准备 Anthropic API Key。

**部署步骤：**

1. 克隆代码到服务器目标目录（例：`/data/project/luxsin-tool`）；
2. 复制 `.env.example` 为 `.env` 并填入真实配置（AI 密钥、模型名等）；
3. 执行 `docker compose up -d --build` 启动服务；
4. 首次启动会自动建表并拉取 Redis 镜像；
5. 访问 `http://<服务器>:8000/` 进入聊天界面。

**`docker-compose.yml` 定义两个服务：**

- `redis`：基于 `redis:7-alpine`，启用健康检查；
- `app`：基于项目 `Dockerfile` 构建，依赖 `redis` 健康后启动，挂载 `./data`（SQLite）与 `./logs`（日志）为宿主机目录，配置日志轮转（单文件 10m、最多 5 份）。

### 5.2 自动化持续部署

项目内置基于 Git Hook 的自动部署：

- `.githooks/post-commit`：开发者本地 commit 后触发后台部署；
- `scripts/deploy.sh`：通过 `rsync` 同步工作区到远端服务器，再 SSH 执行 `docker compose up -d --build`；
- 仅在配置的分支（默认 `main`）上触发，可通过 `SKIP_DEPLOY=1` 环境变量跳过；
- 通过 `Makefile` 暴露常用命令：`make install-hooks`、`make deploy`、`make deploy-log`、`make up-build`、`make logs` 等。

### 5.3 配置项

| 配置项 | 说明 | 默认值 |

## 六、使用说明

### 6.1 日常对话

用户打开浏览器访问系统首页，若局域网内已连接 Luxsin 设备，系统自动识别设备 MAC 并恢复上次未结束的会话。用户可：

- 在输入框输入自然语言问题，按回车或点击"发送"；
- 查看 AI 实时流式响应；
- 点击"+"新建会话；
- 在下拉框中切换历史会话；
- 点击"清空"删除当前会话。

### 6.2 EQ 优化

- 用户自然语言描述听感需求，例如 "低频不够厚" "人声太刺耳"；
- AI 自动读取当前 EQ → 生成优化方案 → 以"优化卡片"形式展示；
- 用户点击 "View comparison" 查看 A/B 曲线；
- 用户点击 "应用" 将优化方案写入设备；
- 用户随时可点击 "回滚" 恢复原参数；
- 历史优化记录可在 "Optimization Records" 面板统一查看。

### 6.3 设备操控

用户可直接说 "把音量调到 50" "切换到蓝牙输入" "创建一个低频增强的 EQ 并保存"，系统会把这类意图翻译为一次或多次设备 API 调用，并在确认危险操作（写入/删除）前弹出模态确认框。

## 七、技术特色与创新点

1. **前后端双通道工具调用**
   首创的"后端直连 AI + 前端直连设备"混合执行模式，既保留了 AI 后端生成能力，又避免了后端对用户家庭网络的穿透需求。借助 Redis Pub/Sub 实现了两种调用通道在对话循环内的统一调度。

2. **多轮自主任务分解**
   通过 `while True` + `stop_reason` 分派的对话编排器，在单次用户输入下让 AI 自主完成多步骤任务（如"读取当前 EQ → 生成优化 → 写入设备 → 反馈结果"），显著提升可用性。

3. **可回滚的 EQ 优化**
   每次优化保存完整的 `before_peq`/`after_peq` 快照和 `applied` 状态，支持随时 A/B 对比、一键回滚，形成了音频调校领域的"版本化"使用范式。

4. **上下文摘要循环**
   针对大模型对话上下文膨胀问题，实现了可配置阈值的异步摘要 + 摘要链路续写机制，在不牺牲对话连贯性的前提下大幅降低长对话的 token 消耗。

5. **消息完整性自愈**
   `repair_messages` 算法可自动修复因网络中断、用户刷新等导致的消息协议残缺，保证 AI 协议合法性，提升系统鲁棒性。

6. **容器化 + 钩子驱动的自动部署**
   轻量化的 `post-commit` Hook + `rsync` + `docker compose` 组合，为小团队提供了开箱即用的持续部署方案。

## 九、安全性设计

1. **密钥隔离**：AI API Key、数据库连接串等敏感配置仅存于 `.env`（未加入版本控制），容器通过 env_file 注入；
2. **操作确认**：所有会改动用户设备的前端工具调用（`set_device_settings`、`set_peq`、`delete_peqs`）均需经过模态对话框二次确认；
3. **网络隔离**：AI 后端不访问用户局域网，设备通信仅限浏览器↔局域网设备直连；
4. **软删除**：会话删除采用标记机制，避免误操作导致数据丢失；
5. **健康检查**：应用与 Redis 均配置健康检查端点，支持容器编排层自动恢复；
6. **超时兜底**：AI 请求、前端工具结果等待均有超时机制，避免协程泄露；
7. **异常上下文捕获**：所有异常均由 `logger.exception` 记录完整堆栈，便于审计。
